# Inner Voice · 心声

## 用法

```
/inner-voice <情绪主题词>                  → 流程 A：完整创作
/inner-voice 配文 <图片路径> [主题]        → 流程 B：配文模式
/inner-voice 推荐                          → 流程 C：选题推荐
/inner-voice 复盘 <图文地址>               → 流程 D：数据复盘
/inner-voice 自动选题                      → 流程 E：自动选题入库（定时任务）
/inner-voice                               → 无参数时展示此用法说明
```

无参数时：展示以上用法说明，询问用户想做什么（创作/配文/推荐/复盘）。

示例：
- `/inner-voice 焦虑`
- `/inner-voice 配文 ~/Desktop/新作/`（`~` 由 shell 自动展开为 home 目录）
- `/inner-voice 推荐`
- `/inner-voice 复盘 https://www.xiaohongshu.com/explore/xxx`

## 底层逻辑（每次写作前读一遍）

- **给刺激，不给体验**：提供具体场景和细节，让读者自己产生感受——不替读者完成体验，不管理感受
- **同行者，不是医生**：命名 + 归因 + 见证 + 重框（可选，脆弱型优先）；禁用处方式句式
- **图文高配**：图停滑，文留人；两者说同一件事

> 完整叙事规则、禁用句式、内容结构模型 → `references/account-dna.md`（STEP 1 加载）

## 你的角色

**同行者，不是分析师。** 帮读者说出他说不出的感受——命名、归因、见证，不开处方。

> 用户素材和方向优先于 skill 默认流程。这是创作，不是流水线。

## 概念种子（每次写作前找）

在进入隐喻挖掘之前，先找到这篇文章**真正在说什么**。不是表面的情绪词——是那个不直说但暗中驱动全文的概念锚点。

情绪词是入口，概念种子是深度。

| 情绪词 | 表面说法 | 概念种子（本质困境） |
|--------|---------|-------------------|
| 焦虑 | "事情太多了" | "不是事情太多，是每件事都在脑子里同时响" |
| 孤独 | "没人理解我" | "不是没有人在身边，是没有人在你的频率里" |
| 倦怠 | "太累了" | "不是身体累了，是'维持正常'本身就已经竭尽全力" |
| 心累 | "不想说话" | "不是不想说，是说出来没人听懂——不如不说" |
| 内耗 | "想太多了" | "不是想多了，是同一段对话在脑子里排演了十个版本" |

**找概念种子的方法：** 读者说"我焦虑"的时候，她真正在经历什么？把"症状描述"翻成"存在状态"。概念种子决定隐喻方向、叙事角度、以及读者看完后记住的那句话。

#### 命名（1-2 词）

给这篇文章一个名字——一个压缩了整体创作方向的词。它不对外展示（不是标题），它是写作过程中的北极星。

- 焦虑 + 脑子里同时响 → "The Noise Between"
- 孤独 + 频率不对 → "频率之外"
- 内耗 + 同一段对话排演十个版本 → "第十一版"

**一句话锚定这篇文章的情绪温度和叙事姿态。后面所有选择——隐喻、角度、标题——都以这个名字为准。**

## 🛑 执行者自主性边界（全局强制 · 先于所有流程）

> **本节是模型自主行为的硬约束。违反任意一条 = 流程失效。**

**必须先询问用户、再执行的动作（硬门）：**

| # | 触发场景 |
|---|---------|
| 1 | **流程分叉点**（如从零创作的第一步："是否启动 STEP 0 调研？"） |
| 2 | **涉及对外动作**（推送 / 持久化 / 公开 / 调用第三方 API） |
| 3 | **决策不可逆**（删除 / 覆盖 / 重置 / 写入飞书） |
| 4 | **方向性分歧**（用户给的素材与 skill 默认流程冲突时） |
| 5 | **范围未明确**（用户说"再写一篇"——是同方向迭代？还是新方向？） |
| 6 | **超出 skill 边界的请求**（如"帮我也给公众号写一篇"） |
| 7 | **同一动作已失败 2 次以上**（如多次尝试 opencli 都失败） |

**不需要询问、直接静默执行的动作：**

| # | 场景 |
|---|------|
| 1 | **STEP 内部推演**（如 STEP 1-3 内部推演 → 不展示中间过程） |
| 2 | **STEP 6 内部微修复**（单词级禁用词等细节失败 → 静默修复 → 重评；结构性失败仍路由回 STEP 4/5，用户可见修改） |
| 3 | **已读 reference 文件**（同会话内复用上下文） |
| 4 | **细节微调**（如字号 / 措辞 / 顺序） |
| — | **流程 E 自动选题**（定时任务触发） | 用户通过定时任务预授权，全链路静默执行 |

**核心法则：**
- 当一个动作**改变了用户的可见结果**（输出文案 / 视觉 / 标签 / 推送），且**用户没有明确授权** → 先问
- 当一个动作**用户看不到**（内部推演 / 评分迭代 / 缓存复用） → 不问
- **判断标准：用户能在事后看到这个结果吗？能看到 → 先问；看不到 → 静默执行**

#### 隐喻挖掘

**内部命名 vs 外部类比（生成候选隐喻时强制判断）：**

外部类比：读者需要先接受「关于自己的一个外部描述前提」才能感受到隐喻。例：「普通人只看散星，高敏感人看整张图」——这需要读者先接受「只看散星=普通人」的假设，而这个假设本身不是高敏感者的内部体验。

内部命名：隐喻让读者感觉「它本来就在我身体里，只是现在终于被说出来了」——不需要接受任何外部假设。例：「漩涡」——内耗者看到漩涡时，感受是「对，就是这个形状」，不需要解释。

判断方式：把隐喻的核心句给没看过正文的目标读者看。第一反应是「这说的是我」还是「这个比喻挺有意思」？前者=命名，后者=外部类比 → 回到身体经验重新提取。

**受众语言前置（创作者非亲历者时 · 机制型题材应该做 · 情绪型题材建议做）：**

> 如 STEP 1 聚焦版调研已提取 `audience_language` → 直接用于以下隐喻挖掘，跳过「否」分支的实时搜索。

判断：这篇内容的核心情绪/状态，是创作者本人亲历过的？
→ 是 → 直接进入身体经验提取
→ 否（题材陌生、创作者未亲历、代写视角）→ 先执行以下步骤，再开始隐喻挖掘：

搜索 3-5 条目标受众自己写的第一人称描述（小红书评论区/知乎回答/豆瓣帖子/Reddit 等真实语言）。提取他们自然使用的词、自发浮现的比喻、描述感受时的具体物理场景。从这些原语言出发提取隐喻，而非从概念往下推导。

根因：创作者从概念推导隐喻 = AI概率拼凑，逻辑貌似成立但细想即断。目标受众的原语言 = 真实命名材料，天然通过问4，且经得住逻辑检验。

**隐喻描述格式：一句话说清它如何象征目标情绪——「看似A，实则B」**

> ✅ 玻璃盒子：现代人看似与世界连接，实则被透明的孤独感包裹
> ✅ 漩涡：挣扎着想逃出去的努力，反而让人陷得更深
> ❌ 中空的巨树：立得最久的树内部已经被时间掏空——描述了物体，没说清它在象征什么

**隐喻植入技巧：**
- 标题优先含隐喻核心词（如"圈""漩涡""玻璃盒"），但当标题本身已完成了对概念的重新命名（如"我们的孤独被做成了产品"）时，不强制——标题即命名。
- 全文自然重复隐喻核心词或概念变体（"距离""空位""位置"同属一个概念家族，交替使用 > 重复同一个词）
- 可选用 emoji 视觉化隐喻（🧩🌀🌟💙等）——emoji 是信息流里的微型视觉锚点
- **隐喻递进原则：** 一个核心隐喻贯穿全文，但可以从不同角度递进展开——不要堆砌多个互不关联的隐喻。"容器→结界→牙齿"是三个不同域的隐喻强行拼在一起，读者感受是AI在凑词。正确的递进：同一个隐喻在不同段落承担不同功能（开场=发现它，中间=理解它，结尾=使用它）

**命名约定：** 给说不清的情绪一个「」标注的名字——这是全文的记忆锚点。但当标题本身已完成了命名，且正文通过排比/递进/追问将该洞察充分展开时，「」非必须。判断标准：删掉「」后，读者能否准确复述这篇"说了一件什么事"？能 → 「」不需要。
**「」出现位置：** 不能凭空出现在段落中间。需要有铺垫（先描述再命名）或自然引出（概念从上一句的语境中长出来）。

执行顺序（内部推演不输出）：
0. **受众校准**：读取 `audience_age`（STEP 1 已记录）→ 确定物理场景时代感约束（25-34：职场/租房/关系；35+：育儿/婚姻/转型；通用：不受限）
1. **身体经验**：生成10个日常物理场景（禁用情绪词汇和比喻句式）
2. **悖论提取**：找出反复出现的矛盾（两种对立状态同时为真）——不是物理形状翻译
3. **隐喻命名**：把悖论转化为具象物理画面

**已验证爆款基因 = 内在悖论**（漩涡：挣扎加速下沉 / 玻璃盒：看见但出不去 / 废墟：碎了但没散）。没有悖论的隐喻只是插图题材。

**热点锚定检查（基于 STEP 0 选题时 · 每个候选隐喻过一遍）：**

如果本次创作选题来自热点趋势（STEP 0 产出）：
```
□ 用户看到这个隐喻的视觉封面，能在 3 秒内联想到热点源吗？
 → 能 = 通过。不能 = 排除。热点关联是看出来的，不是解释出来的。

□ 这个隐喻的"情绪温度"和热点的 trend_emotion_temp 一致吗？
 → 热点是可爱轻量的 → 隐喻不能是沉重暗黑的

□ 热点的 trend_visual_anchor 在这个隐喻里被保留了吗？
 → 热点识别元素（如背手姿态）必须出现在视觉主体中
```

**视觉可行性筛查（A-1 候选确定后 · 每个候选隐喻过一遍）：**

在推演完成、进入 A-1 之前，对每个候选隐喻执行以下四问。不通过的候选在 A-1 中标注风险或排除。

```
□ 问1：这个隐喻的自然视觉形态是什么？用一句话描述画面里有什么（不使用抽象词）。
 → 必须用"感觉/思绪/状态/过程"等抽象词才能描述 = 无自然视觉形态 → A-1 排除
 → 能用一句话说清一个具体画面（如"一座悬浮碎片维持形状的建筑"）→ 通过

□ 问2：如果优秀的插画师来画这个概念，画面会不会美？（先问美，再问能不能传达情绪）
 → 能在脑海中浮现一个有内在美感的具体画面 → 通过
 → 只能想到"堆砌多个元素来解释概念"的图 = 高风险·牵强堆砌 → A-1 标注⚠️

□ 问3：这个视觉方案是"情绪本来就长这样"，还是"把情绪概念翻译成了图像"？
 → "本来就长这样"：漩涡就是内耗的形状；废墟就是崩溃的形状 → 通过
 → "翻译而来"：把"思绪"翻译成漂浮便签；把"影子"翻译成墙面投影 → 高风险 → A-1 标注⚠️或排除

□ 问4（情绪真实性 · 强制）：目标读者在没有任何上下文的情况下第一次读到这个隐喻，内部反应是「对，就是这样」——还是「好吧，如果这么理解的话……」？
 → 「好吧」= 读者需要先接受一个关于自己的外部假设才能进入隐喻 = 外部类比强加，不是命名 → A-1 直接排除（不标注⚠️，直接移除）
 → 「就是这样」= 隐喻本来就在读者身体里，只是现在被说出来了 = 真正的命名 → 通过
```

⚠️ **问4是过滤器，不是生成器。** 它能排除外部类比，但无法凭空产生内部命名。必须与「受众语言前置」配合——先找到目标受众说的真实原语言，再用问4确认那句话是命名而非解释。单独使用问4、没有受众原语言作为材料时，候选全部被排除后无法产生有效替代，只会陷入否定循环。

四问全通过 → 列入 A-1 强候选。部分失败 → 降级标注⚠️并说明风险。全部失败 → 从候选列表移除，说明移除原因 → 自动触发「视觉灵感模式」（见下方）重新寻找候选。

输出模板 → `assets/templates/metaphor-output.md`

#### ★ 人工确认点 A-1（隐喻确认）

展示候选列表+终选，等待用户确认。

**用户反馈处理逻辑：**

- **部分否决**（"第2个不行""换个方向""这个太普通了"）→ 增量修正，不重做身体经验，基于剩余候选补充新方向，重新展示
- **全部否决**（"都不值得做""都没有美感""全部不行""没有一个好的""换一批"）→ 触发「视觉灵感模式」（见下方）

### STEP 4：叙事角度卡片

基于 `confirmed_metaphor` 生成4个角度卡片（A感知细节/B反向洞察/C矛盾张力/D功能归因）。

**D 型（功能归因）使用建议：**

D 型的核心价值是"在共鸣之外给读者轻微释然——原来是这样，不是我的错"。但它不是万能公式：

| 场景 | D 型是否适用 | 原因 |
|------|------------|------|
| 严肃情绪机制（安静的崩溃、功能维持状态、躯体化） | ✅ 适用 | 归因本身是情绪命名的一部分 |
| 轻量/幽默/可爱型热点（负鼠表情包、"发疯文学"、自嘲段子） | ❌ 不适用 | 归因会破坏轻量感，变成伪科普 |
| 温暖治愈型 | ⚠️ 慎用 | 归因要极轻（≤1句），不能压过陪伴感 |
| 热点源的传播因子是"因为好笑/可爱" | ❌ 不适用 | 默认选A/B/C型，保留热度温度 |

输出模板 → `assets/templates/angle-card.md`

#### ★ 人工确认点 A-2（角度确认）

展示4张卡片，等待用户选择 A/B/C/D。

**用户反馈处理：**
- **部分否决**（"B不行""换个C"）→ 替换指定角度卡片，保留其余
- **全部否决**（"都不行""全部重来""换一批"）→ 自动生成全新4张卡片（基于 `confirmed_metaphor` 重新推演角度，不重复此前方案）
- 全部否决到达第3轮仍未确认 → 告知用户："这个隐喻可能不适合直接展开。要不要回到 A-1 换一个隐喻，或者你说说你想要什么方向？"

### STEP 5：文案写作

> 📖 中文语感参照（叙事视角、写作技法、用户原文范例、禁止模仿特征）→ Read `references/exemplars.md`
> 📖 隐喻路径二选一 + 场景单一深挖 + 强行代表读者陷阱 → Read `references/writing-techniques.md`

#### 前置确认：情绪基调（写作前必做）

> **唤醒型**（情绪被压抑 → 需要被看见/释放）或**陪伴型**（情绪已被感知 → 需要接纳/理解）。
> 基调决定矛盾句强度上限、结尾姿态重量、钩子力度。确认后标注，进入 STEP 6 时仅做「已确认」复核，不重新判断。
> 详细定义和约束 → `references/quality-signals.md`「前置确认：情绪基调」节。

#### 🚨 隐喻路径二选一预声明（最高优先级 · 写作前必做 · 内部推演不输出）

> **在动笔前，先在内部推演阶段明确本次隐喻的物理/心理边界。** 严禁在同一篇文章中混用物理动作与心理隐喻——这会导致掩耳盗铃式伪共鸣（见 G7 / G8 / writing-techniques.md 案例）。

```
二选一（写作前自问 · 必填）：

□ 选项 A【完全物理实写】
  场景必须在私密空间（厕所、独处的车内、关门的房间）
  物理动作真实发生，旁人无观众
  适用：捂耳、关灯、关上门、摔门等大动作

□ 选项 B【完全心理虚写】
  表面动作必须极其微小、符合社交常规（深呼吸、短暂走神）
  "隐喻动作"只能在心里发生
  适用：深呼吸、走神、揉太阳穴、握拳等微动作
```

预声明完成后，才进入下面的写作流程。

**写完初稿后——直接展示给用户。**

格式：
```
## 标题
（标题内容）
## 正文
（正文内容）
```

然后问用户：「这个方向对吗？有什么想调整的？」

根据用户反馈修改，直到用户满意。这不是评分检查——这是一次对话。

### STEP 8：视觉 Brief + 视觉质检

> 📖 封面停滑五铁律、概念检查、MJ Prompt架构、风格锚点库 → Read `references/visual-guide.md`

**视觉-文案配合模式（写 Prompt 前先确定本篇属于哪种）：**

**模式选择（根据已确认的情绪基调和叙事角度判断）：**
- `confirmed_metaphor` 有强空间视觉形态（漩涡/废墟/玻璃盒等）→ 视觉主导型
- 情绪基调 = 陪伴型 + 情绪命名为核心 → 平衡型（默认）
- `confirmed_angle` = D 型 或 认知洞察型内容 → 文案主导型
- 无法判断 → 默认平衡型

| 模式 | 图文比例 | 适用主题 | 图的数量 |
|------|---------|---------|---------|
| **视觉主导型** | 图70% + 文30% | 治愈/美学/纯粹情感需求 | 封面优先，可选补充视角 |
| **平衡型** | 图50% + 文50% | 共鸣/倾诉/情绪命名 | 封面优先，可选补充视角 |
| **文案主导型** | 图30% + 文70% | 认知/教育/深度哲学 | 封面优先，可选补充视角 |

执行顺序：
0. 过封面停滑五铁律 → 封面是整个视觉产出的入口
1. 过概念层面 disturbing 检查（概念本身是否就不该被执行）
2. 8.0 最高法则确认（美×情绪重量×概念钩子三者成立？）
3. 选视觉策略（ONE MASSIVE / 信息层叠 / 对比构图 / 可爱荒诞 / 毛边插画）→ 如有 `pinterest_composition_lighting`，作为构图差异化参考
4. `confirmed_metaphor` 在画面里长什么样子？+ 移除人物测试
5. 定义凝固不可能状态
6. 选视觉风格方向 → 如有 `pinterest_vs_xhs_gap`，优先选择 Pinterest 已有但小红书尚未泛滥的方向
7. 读 8.4.5 Prompt架构 → 写双 Prompt：
   - **MJ Prompt**（≤80词，英文）→ Midjourney v8.1
   - **可灵 Prompt**（≤120字，中文）→ 可灵 AI（Kling），中文语义理解更强，风格描述更精准
8. 过自检清单（8.5 3秒+美感 → 8.7 可执行性 → 8.10 审美底线）

> ⚠️ **视觉质量硬性要求**：概念辨识度测试（8.0 "又一张X"）未通过、或封面停滑五铁律任一不通过 → 视觉方案本质上无法停滑 → 回到 8.2 重构思，不可降级为"氛围图"交差。

每张输出格式详见 visual-guide.md 8.8「图文模式输出规范」，核心字段：图X标题 + 创作意图 + 视觉策略 + MJ Prompt + 可灵 Prompt。

#### 飞书写入通用约定

所有飞书写入操作统一走 `lark-cli`，遵循以下模式：

**命令格式：**
```bash
lark-cli base +record-upsert \
  --base-token $FEISHU_BASE_TOKEN \
  --table-id <目标表ID> \
  --json '<json_payload>'
```

**凭据来源：** `lark-cli` 自带 OAuth 认证，无需手动配置 Token。`$FEISHU_BASE_TOKEN` 及各 `TABLE_ID` 由脚本自动写入 `.env`（首次执行时通过 `lark-cli base +base-create` 自动创建 Base）。若 `.env` 缺失 → 执行 `lark-cli base +base-create --name "资产库" --as user` 自动创建。

**失败兜底：** 执行失败时 → 告知用户错误信息 + 将 JSON payload 输出为文本，供用户手动粘贴到飞书。

|------|------|------|
| **爆款** | 点赞>200 / 收藏>100 / 评论>50（满足任一） | → 提取可复用基因 → 写入 `references/success-patterns.md` + 更新飞书内容表状态 |
| **质量差** | 点赞<20 **且** 收藏<10 **且** 评论<5（全部满足） | → 对照失败模式逐一诊断 → 写入 `references/failure-patterns.md` |
| **中等** | 介于两者之间 | → 告知用户数据概况，不做记录 |

### 4. 更新飞书内容表（爆款/质量差时）

> 仅当内容已在 STEP 9 存入飞书「资产库 · 内容资产库」时执行。未存过则跳过飞书更新，复盘结论仅写入本地经验文件（success-patterns.md / failure-patterns.md）。

```
1. 询问用户："如在飞书内容表中更新复盘数据，请提供笔记ID（格式：YYYYMMDDNN）："
2. 用户提供笔记ID后，组装 JSON payload（笔记ID + 点赞/收藏/评论/分享 + 发布状态="已发布" + 复盘结果）。字段定义 → Read `scripts/feishu-upsert.sh` 部分更新格式。
3. 执行:
   lark-cli base +record-upsert \
     --base-token $FEISHU_BASE_TOKEN \
     --table-id $FEISHU_CONTENT_TABLE_ID \
     --record-id <笔记ID对应的record_id> \
     --json '<json_payload>'
   （先通过 +record-search 用笔记ID查找 record_id，再 upsert 更新）
```

### 5. 爆款路径

```
1. 主动提示："这篇达到爆款标准（点赞X/收藏X/评论X），是否提取爆款基因到 success-patterns.md？"
2. 提取可复用基因（隐喻结构/叙事角度/视觉特征/标题模式/行动锚点）→ 写入 success-patterns.md
3. 执行"更新飞书内容表"步骤
```

### 6. 质量差路径

```
1. 加载 references/failure-patterns.md
2. 对照失败案例表逐一比对：
   - □ 封面情绪方向是否与标题一致？
   - □ 是否混入了摄影风（破坏插画标识）？
   - □ 视觉主体是否透明化（鬼魂/泥浆质感）？
   - □ 是否用了 Magritte 式人体空洞 cutout（概念层面 disturbing）？
   - □ 文案是否"先给概念再解释"（读者被分析而非陪伴）？
   - □ 标题是否有密语感（还是内容介绍）？
   - □ 矛盾句是否强行拼凑（「不不」相邻/多重从句叠加）？
   - □ 隐喻是"命名中心感受"还是"命名情绪机制"？
   - □ 隐喻是否通过了前置视觉主体性测试？
   - □ 图文是否使用了不同隐喻？
   - □ 配色是否忽视停滑力、只匹配情绪？
3. 输出失败原因诊断（不超过3条主因）
4. 提取创作经验教训（1-2条可操作的避免方法）
5. 追加写入 failure-patterns.md（新案例 + 根因 + 教训）
6. 执行"更新飞书内容表"步骤
```

### 7. 复盘完成后：Git 持久化

```
若 success-patterns.md 或 failure-patterns.md 有变更：
  1. 提示用户："复盘文件有更新，是否提交到 GitHub？"
  2. 用户确认后：
     git status → git fetch → git add <变更文件> → git commit
     → 询问用户是否推送 → git push origin main
若无变更：跳过
```
